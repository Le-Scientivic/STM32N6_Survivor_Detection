04 — Pipeline d'inférence

Trajet d'une frame du capteur caméra à l'affichage des survivants détectés

🎬 Vue synthétique

Une frame est acquise par le capteur, dérivée en deux flux par le DCMIPP, traitée par le NPU, post-traitée par le CPU, puis projetée sur l'écran avec les boîtes englobantes superposées. Toute cette chaîne tourne à 20–30 FPS grâce au parallélisme entre les 4 tâches du pipeline.

4.1 Schéma du flux de données

Capteur MIPI CSI-2 (IMX335 / VD55G1 / VD66GY)
          │
          ▼
    DCMIPP — Dual Pipe
   ┌───────┴───────┐
   ▼               ▼
┌──────────┐  ┌─────────────────┐
│  Pipe1   │  │  Pipe2          │
│ 800×480  │  │ 480×480 RGB888  │
│ RGB565   │  │ (NN input)      │
└────┬─────┘  └────────┬────────┘
     │                 │ free → ready (sémaphores)
     │                 ▼
     │           ┌──────────────┐
     │           │  nn_thread   │  LL_ATON_RT_Run_Inference (NPU)
     │           └────────┬─────┘
     │                    │ free → ready
     │                    ▼
     │           ┌──────────────┐
     │           │  pp_thread   │  app_postprocess_run() + tracker
     │           └────────┬─────┘
     │                    │ disp.lock + disp.update
     │                    ▼
     │           ┌──────────────┐
     └──────────►│  dp_thread   │  ScreenL : LAYER_0 (caméra) + LAYER_1 (boîtes)
                 └────────┬─────┘
                          ▼
                       LCD 800×480

4.2 Étape 1 — Acquisition caméra

Le capteur produit en continu des frames qui sont injectées dans le DCMIPP. La fonction CAM_Init() (app_cam.c) configure les deux pipes à partir des macros de app_config.h :

// Pipe1 — Affichage 800×480 RGB565 16 bpp
DCMIPP_PipeInitDisplay(LCD_BG_WIDTH, LCD_BG_HEIGHT);

// Pipe2 — Réseau 480×480 RGB888_YUV444_1 3 bpp
DCMIPP_PipeInitNn(NN_WIDTH, NN_HEIGHT, NN_FORMAT);

L'événement VSYNC du DCMIPP (callback app_main_pipe_vsync_event()) signale le sémaphore isp_sem, ce qui réveille la tâche isp_thread. Celle-ci appelle CAM_IspUpdate() pour recalculer en continu les paramètres ISP (balance des blancs, exposition, gamma).

4.3 Étape 2 — Capture du buffer d'entrée NN

Lorsque le DCMIPP a écrit une frame complète dans le Pipe2, le callback app_ancillary_pipe_frame_event() est invoqué. Il transfère le buffer de la zone free vers la zone ready de la file nn_input_queue :

// Schéma : 2 buffers circulant entre l'ISR et la tâche NN
nn_input_queue {
    void *buffers[2];          // 480×480×3 = 691 200 octets chacun
    sem_id  free, ready;       // 2 sémaphores de synchronisation
}

Cette structure de buffer queue garantit zéro copie mémoire : le NPU lit directement le buffer rempli par le DCMIPP.

4.4 Étape 3 — Inférence sur le NPU

La tâche nn_thread s'éveille dès qu'un buffer est ready, puis exécute les appels NeuralART suivants :

ÉtapeAPI NeuralARTDescription
1LL_ATON_RT_RuntimeInit()Initialisation du runtime (au boot)
2LL_ATON_RT_Init_Network(&NN_Instance_Default)Charge la topologie YOLO-X
3LL_ATON_Set_User_Input_Buffer()Pointe le buffer DCMIPP comme entrée
4LL_ATON_RT_RunInference_*()Lance l'exécution sur le NPU
5LL_ATON_Get_User_Output_Buffer()Récupère le tenseur de sortie

Les structures de réseau (NN_Instance_Default) sont déclarées par la macro LL_ATON_DECLARE_NAMED_NN_INSTANCE_AND_INTERFACE(Default) et générées par l'outil de quantification ST. Le tenseur de sortie brut est ensuite poussé dans nn_output_queue.ready.

4.5 Étape 4 — Post-traitement

La tâche pp_thread consomme le buffer de sortie et applique le wrapper de post-processing approprié :

app_postprocess_init();      // une seule fois au boot
app_postprocess_run(nn_out, &pp_out);   // par frame

Selon la macro de compilation choisie dans postprocess_conf.h, le wrapper appelle :

  • POSTPROCESS_OD_ST_YOLOX_UF sur la cible STM32N6570-DK ;
  • POSTPROCESS_OD_YOLO_V2_UF sur les cibles secondaires.

Le post-processing produit une structure od_pp_out_t contenant le nombre de détections (nb_detect) et un tableau de boîtes englobantes pOutBuff[]. Les seuils suivants filtrent les résultats :

MacroValeurEffet
AI_OD_ST_YOLOX_PP_CONF_THRESHOLD0.6Seuil de confiance minimum
AI_OD_ST_YOLOX_PP_IOU_THRESHOLD0.5Seuil d'IoU pour la NMS
AI_OD_ST_YOLOX_PP_MAX_BOXES_LIMIT10Nombre max de boîtes après NMS

4.6 Étape 5 — Tracking (optionnel)

Si la macro TRACKER_MODULE est définie à la compilation, le post-processing est suivi d'un appel au tracker :

#ifdef TRACKER_MODULE
trk_update(&trk_ctx, &pp_out);   // attribution d'un ID persistant
#endif

Le tracker maintient un identifiant stable par survivant à travers les frames, à l'aide d'un filtre de Kalman et de seuils :

  • track_thresh = 0.25 — seuil d'association
  • det_thresh = 0.8
  • sim1_thresh = 0.8 · sim2_thresh = 0.5
  • tlost_cnt = 30 — nombre de frames sans détection avant perte de l'ID

4.7 Étape 6 — Synchronisation et affichage

Une fois les détections finalisées, pp_thread écrit le résultat dans la structure partagée disp.info, sous la protection du mutex disp.lock, puis signale le sémaphore disp.update.

La tâche dp_thread est alors réveillée. Elle effectue trois opérations :

  1. Lecture protégée de disp.info (mutex disp.lock).
  2. Mise à jour du buffer de dessin (dp_update_drawing_area) avec les rectangles colorés via UTIL_LCD_* et UTIL_LCDEx_PrintfAt (FPS, charge CPU).
  3. Bascule de buffer (dp_commit_drawing_area) — clean cache + swap pointeur.

4.8 Mesure de performance

Chaque tâche horodate son entrée et sa sortie. Les valeurs sont accumulées dans la structure disp.info :

ChampMesure
inf_msDurée d'inférence NPU
pp_msDurée du post-processing + tracker
disp_msDurée du rendu LCD
nn_period_msPériode entre deux inférences (= 1/FPS)
Charge CPUMoyenne sur 8 ticks (cpuload_info_t dans utils.c)

Ces valeurs sont à la fois affichées sur l'écran et émises sur la console série au format JSON (cf. page suivante).
