03 — RTOS μT-Kernel 3.0

Multitâche préemptif, sémaphores, mutex et files de buffers

🧩 μT-Kernel 3.0 BSP2

Le projet utilise le noyau temps réel μT-Kernel 3.0 BSP2 sous T-License 2.2, un RTOS léger conçu pour les systèmes embarqués contraints. L'intégration STM32Cube est effectuée par mtkernel_bsp.c, et le code du noyau est hébergé sous Appli/mtk3_bsp2/.

3.1 Configuration du noyau

Les paramètres clés du noyau sont définis dans Appli/mtk3_bsp2/config/ :

ParamètreValeurEffet
CNF_MAX_TSKID32Nombre maximal de tâches simultanées
CNF_MAX_SEMID16Nombre maximal de sémaphores
CNF_MAX_MTXID8Nombre maximal de mutex
CNF_TIMER_PERIOD1 msPériode du tick noyau
Pile temporaire256 octetsPile interne des appels système
USE_IMALLOC1Allocation dynamique activée
SYSTEM_MEM_SIZE10 KBPool mémoire système statique

3.2 Architecture des tâches

L'application instancie 5 tâches qui s'exécutent en parallèle. Toutes utilisent la même taille de pile (2 × UTKERNEL_STACK_SIZE = 4096 octets) ; les priorités sont numérotées en sens inverse (plus le numéro est petit, plus la priorité est haute).

TâchePrioritéPileRôle
isp_thread10 (la + haute)4096 oMise à jour de l'ISP caméra (déclenchée par VSYNC)
nn_thread114096 oInférence YOLO-X sur le NPU
pp_thread144096 oPost-traitement (NMS) et tracking
dp_thread144096 oAffichage : dessin des boîtes englobantes
main_thread15 (la + basse)4096 oInitialisation et supervision
⚠️ Pourquoi cette hiérarchie

L'ISP est prioritaire pour ne jamais perdre de frame caméra. Le NN est juste en dessous, car son temps d'exécution (~30–50 ms) doit s'absorber sans accumulation de retard. Le post-processing et l'affichage tolèrent davantage de jitter et partagent donc la priorité 14.

3.3 Création des tâches

Toutes les tâches sont créées dans app_run() par appels successifs à tk_cre_tsk() puis démarrées par tk_sta_tsk() :

// Pseudocode (extrait simplifié de app.c)
T_CTSK ctsk = {
    .stksz = 2 * UTKERNEL_STACK_SIZE,
    .tskatr = TA_HLNG | TA_RNG3,
};

ctsk.task = nn_thread_fct;  ctsk.itskpri = 11;
nn_tskid = tk_cre_tsk(&ctsk);
tk_sta_tsk(nn_tskid, 0);

ctsk.task = pp_thread_fct;  ctsk.itskpri = 14;
pp_tskid = tk_cre_tsk(&ctsk);
tk_sta_tsk(pp_tskid, 0);

ctsk.task = dp_thread_fct;  ctsk.itskpri = 14;
dp_tskid = tk_cre_tsk(&ctsk);
tk_sta_tsk(dp_tskid, 0);

ctsk.task = isp_thread_fct; ctsk.itskpri = 10;
isp_tskid = tk_cre_tsk(&ctsk);
tk_sta_tsk(isp_tskid, 0);

3.4 Services RTOS utilisés

Gestion des tâches

  • tk_cre_tsk() — création
  • tk_sta_tsk() — démarrage
  • tk_slp_tsk() — endormissement
  • tk_wup_tsk() — réveil
  • tk_dly_tsk() — délai

Synchronisation

Le projet définit cinq objets de synchronisation explicites :

ObjetTypeRôle
nn_input_queue.free / .readySémaphoresFile circulaire à 2 buffers d'entrée NN
nn_output_queue.free / .readySémaphoresFile circulaire à 2 buffers de sortie NN
isp_semSémaphoreSignalé par l'ISR VSYNC pour réveiller isp_thread
disp.updateSémaphoreNotifie dp_thread qu'un nouveau résultat est prêt
disp.lockMutex (FIFO)Protège la structure partagée disp.info entre pp_thread et dp_thread

Mesure du temps

  • tk_get_otm() — horloge système RTOS pour profilage
  • HAL_GetTick() — tick milliseconde STM32 HAL

3.6 Bénéfices apportés par le RTOS

  1. Déterminisme — la priorité préemptive garantit que la caméra et le NPU obtiennent du CPU à temps.
  2. Parallélisme du pipeline — acquisition, inférence, post-traitement et affichage s'exécutent simultanément.
  3. Synchronisation propre — sémaphores et mutex remplacent toute approche par flags globaux fragiles.
  4. Gestion mémoire — les buffers sont alloués statiquement, sans fragmentation.
  5. Réactivité — l'ISR VSYNC ne fait que signaler un sémaphore, le travail réel est différé dans isp_thread.
