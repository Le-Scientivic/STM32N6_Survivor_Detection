STM32N6 Survivor Detection

Modèle IA — YOLO-X Nano

Réseau de détection d'objets quantifié INT8, optimisé pour le NPU STM32N6 par STMicroelectronics

🧠 Choix du modèle

Le projet utilise YOLO-X Nano, une variante allégée et optimisée par ST de la famille YOLO-X. Elle a été choisie pour son ratio précision / latence sur NPU embarqué : un seul réseau exécuté entièrement sur l'accélérateur, à 25–30 FPS, pour une consommation mémoire compatible avec la flash externe.

2. Génération du modèle (NeuralART)

Le modèle TFLite est compilé pour le NPU STM32N6 par l'outil NeuralART de STMicroelectronics. La configuration est définie dans Model/STM32N6570-DK/user_neuralart.json :

{
  "memory_pool": "./my_mpools/stm32n6-app2.mpool",
  "options": "--enable-epoch-controller -O3 --all-buffers-info --mvei
              --cache-maintenance --Oalt-sched --native-float
              --enable-virtual-mem-pools --Omax-ca-pipe 4
              --Ocache-opt --Os"
}
Option NeuralARTEffet
--enable-epoch-controllerActivation du contrôleur d'époques (planification du NPU)
--cache-maintenanceInsertion des invalidations / clean cache nécessaires
--Oalt-schedScheduler alternatif optimisé

4. Intégration côté firmware

Le firmware référence le réseau via la macro de déclaration NeuralART, puis manipule l'instance comme un objet opaque :

// Déclaration de l'instance et de son interface
LL_ATON_DECLARE_NAMED_NN_INSTANCE_AND_INTERFACE(Default);

// Initialisation (au boot, dans app_run)
LL_ATON_RT_RuntimeInit();
LL_ATON_RT_Init_Network(&NN_Instance_Default);

// Cycle d'inférence (dans nn_thread)
LL_ATON_Set_User_Input_Buffer(&NN_Instance_Default, input_buf, NN_BUF_SIZE);
LL_ATON_RT_RunInference_*(&NN_Instance_Default);
output = LL_ATON_Get_User_Output_Buffer(&NN_Instance_Default);

5. Post-traitement

La sortie brute du réseau est un tenseur de prédictions YOLO (boîtes + scores + classes). Le wrapper app-postprocessing-wrapper standardise cette sortie en une liste de boîtes filtrées par NMS et seuil de confiance, accessible via la structure od_pp_out_t.

typedef struct {
    int        nb_detect;       // Nombre de boîtes après NMS
    od_box_t  *pOutBuff;        // Tableau de boîtes
    /* class, score, x, y, w, h pour chaque détection */
} od_pp_out_t;

Paramètres de filtrage

6. Tracking optionnel

Lorsque la macro TRACKER_MODULE est définie, un module de tracking attribue un identifiant persistant à chaque survivant détecté. Le contexte trk_ctx conserve l'historique des détections et applique un filtre de Kalman entre frames.

8. Adapter le modèle

Pour modifier la sensibilité ou la cadence du système, il suffit en général d'ajuster app_config.h sans regénérer le modèle :

  • Augmenter AI_OD_ST_YOLOX_PP_CONF_THRESHOLD → moins de faux positifs, plus de faux négatifs.
  • Diminuer AI_OD_ST_YOLOX_PP_IOU_THRESHOLD → la NMS supprime davantage de doublons.
  • Modifier NB_CLASSES → impose une régénération du modèle TFLite avec le bon nombre de classes.
⚠️ Régénération du modèle

Toute modification de la résolution d'entrée, du nombre de classes ou de l'architecture impose de relancer generate-n6-model.sh et de reprogrammer la flash NOR avec le nouveau network_data.hex.
