Architecture Matérielle

Plateforme STM32N6570-DK Discovery — MCU, NPU, mémoires externes, capteurs caméra et écran LCD

📐 Vue d'ensemble

Le système repose sur la carte de développement STM32N6570-DK de STMicroelectronics. Toute la chaîne d'acquisition vidéo, d'inférence IA et d'affichage est exécutée sur la même puce STM32N657, qui combine un cœur Arm Cortex-M55 et une unité de traitement neuronal (NPU) accélérant les réseaux de convolution.

2. Mémoires

Le projet exploite plusieurs régions mémoires distinctes, dont l'organisation est imposée par la séparation FSBL / application et par la taille du modèle IA.

Activation des mémoires externes

Les bus XSPI sont initialisés au tout début de la tâche principale (main_thread_fct) :

BSP_XSPI_RAM_Init(...);   // RAM externe en mode mémoire
BSP_XSPI_RAM_EnableMemoryMappedMode(...);

BSP_XSPI_NOR_Init(...);   // NOR flash en mode OPI / DTR
BSP_XSPI_NOR_EnableMemoryMappedMode(...);

3. Pipeline caméra (DCMIPP + MIPI CSI-2)

L'acquisition vidéo utilise l'interface MIPI CSI-2 deux lignes couplée au contrôleur DCMIPP (Digital Camera Memory Interface Pixel Pipeline). Le DCMIPP permet de dériver deux flux à partir d'une même source caméra :

Pipe DCMIPPRésolutionFormatDestination
Pipe1 — Display800 × 480RGB565 (16 bpp)Écran LCD via LTDC
Pipe2 — NN input480 × 480RGB888_YUV444_1 (3 bpp)Buffer d'entrée du NPU

Capteurs caméra supportés

Le middleware caméra CMW abstrait trois capteurs MIPI CSI-2 :

  • Sony IMX335 — capteur 5 MP, profil ISP dans imx335_isp_param_conf.h
  • STMicroelectronics VD55G1 — défini par USE_VD55G1_SENSOR
  • STMicroelectronics VD66GY (VD6G) — profil ISP dans vd66gy_isp_param_conf.h
⚠️ Cadence d'acquisition

La fréquence de capture est fixée à CAMERA_FPS = 20 sur la cible STM32N6570-DK (déclaration dans app_cam.h). L'orientation est contrôlée par CAMERA_FLIP = CMW_MIRRORFLIP_MIRROR.

4. Affichage LCD via LTDC

L'écran intégré à la carte (800×480, RGB565) est piloté par le contrôleur LTDC. Le projet utilise la bibliothèque ScreenL (SCRL), qui propose deux couches superposées :

  • LAYER_0 — couche de fond, recevant le flux caméra Pipe1 (RGB565).
  • LAYER_1 — couche d'overlay au format ARGB4444, sur laquelle sont dessinées les boîtes englobantes des survivants détectés et les informations de monitoring (FPS, charge CPU).

Un mécanisme de double buffer assure le rendu sans tearing : la tâche d'affichage écrit dans une zone tampon (dp_update_drawing_area) puis l'échange par bascule de pointeur (dp_commit_drawing_area).

6. Sécurité et boot

Le STM32N6 démarre obligatoirement par un FSBL (First Stage Boot Loader) signé, qui s'exécute dans la SRAM interne AXISRAM2. Ce FSBL configure les blocs de sécurité (TrustZone, RIF, IAC), puis transfère l'exécution à l'application principale chargée en RAM externe XSPI1.

┌─────────────────┐
│   Power-On      │
└────────┬────────┘
         ▼
┌─────────────────┐
│  FSBL (Secure)  │  Init HW · sécurité · saut application
│  AXISRAM2       │
└────────┬────────┘
         ▼
┌─────────────────┐
│  Application    │  μT-Kernel + pipeline caméra/IA/affichage
│  XSPI RAM       │
└─────────────────┘

Les configurations de sécurité sont appliquées dans main() via Security_Config() (RIF RIMC/RISC pour NPU, DMA2D, DCMIPP, LTDC, OTG) et IAC_Config().
